# OAuth Path Comparison: cc-lb vs better-ccflare

Reference inspected: `reference/better-ccflare`

Note: the requested path `references/ccflare` does not exist in this workspace. The local reference tree is `reference/better-ccflare`.

Five subagents inspected the current project and the reference project with the same prompt. This document merges the overlapping findings and keeps the meaningful differences.

## Summary

Current `cc-lb` is narrower and simpler: Anthropic Claude OAuth only, tRPC dashboard mutations, strict `code#state` parsing, server-side OAuth state validation, and account-ID-bound reauth.

`better-ccflare` is broader and more operationally complete: REST OAuth endpoints, API-key/RBAC auth, multiple provider/device flows, Anthropic console/API-key mode, per-account custom endpoints, richer structured errors, and broader tests.

The most important security split is:

- Current project stores OAuth `state` and verifies it locally before token exchange.
- `better-ccflare` generates a `state` value in the auth URL, but the inspected session storage does not persist or verify it locally.
- Current reauth sessions bind to account ID; `better-ccflare` reauth callback resolves by account name.
- Current proxy `/v1/*` is unauthenticated by application auth; `better-ccflare` can protect proxy routes with API keys.

## High-Signal Risks If Porting From Reference

1. Do not adopt public `/api/oauth*` exemptions without also keeping local state validation and strong session binding.
2. Keep account-ID-bound reauth rather than resolving by account name.
3. If adding REST compatibility, decide whether OAuth should stay dashboard-auth gated or become public like the reference.
4. If adding console/API-key mode, extend schema/account routing/header logic together; current account model assumes OAuth tokens.
5. Document `CLIENT_ID` and `ANTHROPIC_API_BASE` if they remain supported.
6. Consider adding proxy API-key auth before exposing `/v1/*` outside localhost/private networks.
